# System Instructions & Architecture

JARVIS V2 is built around a hybrid routing model designed to preserve data privacy without sacrificing the advanced reasoning capabilities of cloud models.

## How Routing Works

1. **The Cloud Librarian**: When you send a message to JARVIS, it first goes to a highly capable cloud model (e.g., Gemini 1.5 Pro). This model is strictly configured as a "Librarian." Its system prompt explicitly forbids it from answering the user directly or writing code. 
2. **Intent Analysis**: The Librarian analyzes your request and decides:
   - What capability is needed? (`reasoning`, `coding`, `os`, `general`)
   - What action should be taken? (`save_script`, `run_saved_script`, `run_daemon_script`, `search_web`, `reply_to_user`, `spawn_agents`, etc.)
   - It outputs this decision as a strict JSON map.
3. **Local Fleet Execution**: The Node.js backend catches this JSON map and uses the `capability` field to determine which local Ollama model to wake up.
   - For `reasoning`, it looks for a model like `gemma`.
   - For `coding`, it looks for `qwen`.
   - For `os`, it looks for `hermes`.
4. **Action Execution**: The selected local model is then given the detailed instructions generated by the cloud model and executes the requested action. All file creation, script execution, and memory retrieval happen entirely on your local machine.

## Background Daemons

JARVIS includes a **Background Task Manager** (`src/daemon/manager.js`). This allows JARVIS to launch scripts that run continuously in the background (e.g., while-loops that monitor USB devices, directories, or system health).
- Daemons run completely decoupled from the main event loop.
- Their `stdout` and `stderr` are piped through WebSockets back to the UI in real-time.
- You can manage and kill these tasks from the "Active Daemons" tab in the UI.

## Memory Systems

JARVIS has two distinct memory systems that operate locally:
1. **RAG (Retrieval-Augmented Generation)**: Uses the `embedding-001` model to map conversations into vector space. When you ask a question, JARVIS searches this vector database to pull in relevant past context.
2. **Core Memory (Vault)**: A structured text file containing hard facts about the user and the system. This is automatically updated when JARVIS detects new, important information.

## Multi-Agent "Spawn" Mode

If the Cloud Librarian determines your request is highly complex and requires multiple distinct steps (e.g., "Research top 5 AI frameworks and text me the summary on WhatsApp"), it will trigger the `spawn_agents` action.
- The Cloud Planner will decompose your goal into a JSON array of sequential tasks.
- A local Supervisor will spin up specialized local models (e.g., a researcher, a coder, and a messenger) to execute these tasks in order, passing the output of one task as the input to the next.
